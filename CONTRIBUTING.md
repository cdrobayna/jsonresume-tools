# Contributing

This covers how changes flow through the repo — branching, changesets, and how releases
happen. For environment setup (Node/pnpm versions, build/test commands), see the
[Development section of the README](./README.md#development).

## Branching

- `main` is the only long-lived branch. There's no `develop` or `release/*` branch — releases
  are cut directly from `main` via the automated pipeline described below.
- Work in short-lived branches off `main`, named `type/short-description` (e.g.
  `fix/lint-date-regex`, `feat/tailor-cli-flag`, `chore/release-workflow`).
- Open a PR into `main` rather than pushing directly, even for small changes. `main` is what
  triggers the release workflow (see below), so keeping it gated behind CI + review avoids a
  broken package landing there.
- Merge PRs with a merge commit (GitHub's "Create a merge commit", not squash or rebase).
  This keeps each branch's real commits as actual ancestors of `main`, so a stacked branch
  (one PR built on top of another, unmerged one) can sync against the new `main` with a plain
  `git rebase main` once its base lands — no conflicts, no `rebase --onto`/force-push
  surgery. Squash and rebase-merge both mint new commit hashes for the same content, which
  breaks that ancestry and makes any branch stacked underneath show phantom conflicts against
  `main` (same change, unrelated history) even though nothing actually conflicts.
- Delete the branch after merging.
- Never push directly to the `changeset-release/main` branch — it's owned by the release
  Action and gets regenerated on every push to `main`.

## Adding a changeset

Any change that affects a **published** package (`jsonresume-lint`, `jsonresume-parity`,
`jsonresume-tailor`) needs a changeset:

```bash
pnpm changeset
```

This prompts you to:

1. select which package(s) changed,
2. pick a bump type per package — `patch` (fix), `minor` (backwards-compatible feature),
   `major` (breaking change),
3. write a one-line summary — this becomes the CHANGELOG entry.

Commit the generated `.changeset/*.md` file together with your code change, in the same PR.

Changes that don't affect a published package's behavior (docs, CI config, internal
refactors, tests) don't need a changeset.

`@jsonresume-tools/core` is private and never published — you generally don't need to select
it directly; if a published package depends on it, changesets bumps that dependency for you
(`updateInternalDependencies: patch` in `.changeset/config.json`).

## What CI runs

- `ci.yml`: on every push and PR to `main`, builds and runs the test suite on Node 22.x and
  24.x.
- `release.yml`: on every push to `main` only — never on PRs, so `NPM_TOKEN` is never exposed
  to a pull request's context:
  - if there are pending changesets, opens/updates a **"Version Packages"** PR with the
    version bumps and CHANGELOG entries;
  - if there are no pending changesets (i.e. right after that PR is merged), runs
    `pnpm release` (`changeset publish`), which publishes any package whose `package.json`
    version is ahead of what's on npm, and tags/creates a GitHub Release for it.

## Cutting a release

1. Merge PRs with changesets into `main` as normal — the "Version Packages" PR keeps itself
   up to date, accumulating every pending changeset.
2. When you're ready to ship, review and merge the "Version Packages" PR. That merge is what
   actually triggers `npm publish` — there's no manual `npm publish` / `pnpm publish` step.

## What not to do

- Don't hand-edit package `version` fields or `CHANGELOG.md` — they're generated by
  `changeset version`.
- Don't run `npm publish` / `pnpm publish` locally — publishing only happens through the
  release workflow.
- Don't merge the "Version Packages" PR unless you actually want to publish everything it
  contains.
